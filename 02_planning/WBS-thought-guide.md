# Work Breakdown Structure (WBS) Thought Guide  
*A reasoning tool for decomposition, clarity, and scope alignment*

---

## Purpose of This Guide
A Work Breakdown Structure (WBS) is more than a hierarchy of tasks — it is a thinking process.  
This guide helps project managers reason through decomposition, identify blind spots, and ensure the WBS fully represents the project scope before scheduling, estimation, or resource planning begins.

Use this guide during:
- early planning  
- team brainstorming  
- scope clarification  
- WBS refinement  
- pre‑execution alignment  

---

## 1. Foundational Questions  
Start with these prompts to clarify the project’s structure and outcomes.

- What are the **major deliverables** of this project?  
- What does “**done**” look like for each deliverable?  
- What outcomes must exist for the project to succeed?  
- What work must be completed to produce each outcome?  
- What milestones naturally emerge from the project’s goals?  
- What constraints shape the work (time, budget, tools, compliance)?  
- What assumptions are we making about how the work will unfold?

---

## 2. Decomposition Prompts  
Use these questions to break deliverables into actionable tasks.

- Can this deliverable be broken down further?  
- Is this task **actionable**, or is it still too vague?  
- Does each task begin with a **verb**?  
- Is each work package small enough to estimate?  
- Does each work package represent a **single unit of work**?  
- Are we mixing “activities” with “deliverables”?  
- Are we decomposing until the work is **assignable** to one owner?

If you cannot estimate or assign a task, it is not decomposed enough.

---

## 3. Blind Spots & Common Pitfalls  
These are the most frequent issues PMs encounter when building a WBS.

### **Deliverables that aren’t deliverables**
- “Planning,” “Meeting,” “Coordinating,” “Reviewing”  
These are activities — not outcomes.

### **Tasks that hide multiple steps**
- “Prepare materials”  
- “Set up environment”  
These need decomposition.

### **Tasks that are too large**
If a task spans multiple days or requires multiple people, it is likely a work package that needs breaking down.

### **Missing cross-functional work**
Projects often overlook:
- IT support  
- Procurement  
- Legal review  
- Communications  
- Training  
- Change management  

### **Dependencies not represented**
A WBS is not a schedule, but missing dependencies often indicate missing tasks.

### **Overlooking validation steps**
Every deliverable should include:
- review  
- approval  
- acceptance criteria  

### **Not decomposing non-technical work**
PMs often forget:
- stakeholder alignment  
- documentation  
- onboarding  
- rollout support  

---

## 4. Quality Checks  
Use this checklist to validate your WBS before moving into scheduling.

- Does the WBS cover **100% of project scope**?  What's missing?
- Are there any orphan tasks that don’t roll up to a deliverable?  
- Are all tasks written as **verb-based actions**?  
- Are work packages small enough to estimate?  
- Are acceptance criteria clear for each deliverable?  
- Are dependencies implied or missing?  
- Does the hierarchy make logical sense?  
- Would a new team member understand the structure?  
- Does each work package have a single owner?  
- Are assumptions and constraints documented?

---

## 5. PM Judgment Prompts  
These prompts help you think like a senior PM.

- What risks emerge from the structure of this work?  
- Which work packages are most likely to slip?  
- What tasks depend on external teams or vendors?  
- What tasks require specialized skills or approvals?  
- What tasks have unclear ownership?  
- What tasks have unclear acceptance criteria?  
- What tasks might be underestimated?  
- What tasks might be overestimated?  
- What tasks are sensitive to timeline compression?  
- What tasks require early stakeholder engagement?

### **Legal, Regulatory, and Compliance Judgment Prompts**  
- Have we identified any deliverables or tasks that require **legal review**, **regulatory approval**, or **compliance validation**?  
- Are there any **laws, regulations, or industry rules** that shape how this work must be performed?  
- Did we include tasks for:  
  - legal consultation  
  - compliance verification  
  - regulatory research  
  - documentation required by governing bodies  
- If we skipped legal/regulatory review, what structural gaps does that create in the WBS?  
- Could missing legal tasks create downstream risks such as:  
  - project delays  
  - fines or penalties  
  - invalidation of deliverables  
  - forced rework  
  - inability to deploy or launch  
- Do any work packages require **attorney sign-off**, **policy review**, or **risk mitigation** before execution?  
- Are we assuming compliance without verifying it?  
- Are we relying on tribal knowledge instead of documented legal guidance?  
- Does the WBS reflect the **full compliance lifecycle**, not just the build?
  
---

## 6. Versioning & Evolution  
A WBS is a **living artifact**. It evolves as you learn more about the project.

Use these prompts to manage updates:

- Has new information changed the scope?  
- Have stakeholders added or removed deliverables?  
- Have constraints changed (budget, timeline, tools)?  
- Does the WBS still reflect the project’s current reality?  
- Do we need to re-estimate work packages?  
- Do we need to reassign ownership?  
- Do we need to update dependencies?  
- Do we need to revise acceptance criteria?

Update the WBS whenever:
- scope changes  
- new risks emerge  
- new tasks are discovered  
- deliverables shift  
- stakeholders request changes
